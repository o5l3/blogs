## kim-tigerj — Agent sprocess 1시간 hourly 집계 체계 완성

**sprocess 1시간 단위 hourly 집계 체계 완성 + stall 회귀 격리 (OR-1340)**. OR-1306이 깐 1시간 주기 송신 인프라 위에서 송신 수치의 의미를 "부팅 후 누적"에서 "1시간 단위 집계"로 교체했다. 핵심은 `CSProcessHourly` 클래스 하나에 SUID 기반 1시간 버킷 메모리 집계·`Rotate` 공통 진입점·`Cleanup`·`FlushAll`·`DebugFlush`(dev 모드)를 모두 모으고, 절전·시계 변경·VM 재기동처럼 시간이 갑자기 건너뛰는 모든 상황을 `Rotate` 헬퍼가 흡수하도록 통합한 것이다. 모든 진입점이 같은 함수를 거치니 시간 점프 처리 누락이 원천 차단된다.

데이터 모델도 같이 정리했다. `sprocess`는 부팅 누적(실시간), `sprocess_hourly`는 시간 집계로 양 테이블 역할을 분리하고 서버 전송은 hourly만으로 확정. `Running`은 저장 컬럼을 모두 제거하고 8개 `SELECT`에서 `MAX(Start − Stop, 0) AS Running` 파생으로 전환해 0 복귀가 자동으로 따라온다. `sprocess_hourly`의 `Running`은 `MAX(Start2 + Start − Stop, 0)`. `Rotate` 시간대 전환 시 `new.start2 = MAX(old(start2+start-stop), 0)`로 시작값을 자동으로 넘기니, 한 시간대의 마지막 Running이 다음 시간대의 시작값이다. 28시간 무중단 운영에서 40회 연속 시간대 전환 carry mismatch 0건.

이번 주의 가장 어려웠던 부분은 stall 회귀 진단이다. `sprocess` 실시간 누적을 위해 5개 증상 카운터(`Kill/Killed/Crash/Crashed/Detect`)에 신규 `SPROCESS_UPDATE_*` 쿼리 4개를 추가한 직후, `summary.cdb` 메인 파일 쓰기가 1시간 동안 멈추는 회귀가 나타났다. Agent 자체는 정상 동작하지만 외부 reader(`orangedb.exe`)에서 라이브 데이터가 보이지 않는다. 4개 신규 콜 사이트를 `#if 0`으로 임시 격리하고 28시간 검증한 결과 stall이 사라져, lock contention 가설을 확정했다. `wal_autocheckpoint = 1` 노브 신설로 매 commit마다 WAL → 메인 파일이 즉시 flush되도록 `db/summary.json` 헤더에 박았다. dev 모드(`C:@DEV`)에선 `transaction=0` + `wal_autocheckpoint=1`을 강제로 적용해 매 이벤트마다 디스크 반영이 즉시 보이게 만들었다. `SystemStat` MQTT 송신 빈도 폭증(평상시 60초 / LiveResponse 5초)도 같은 작업에서 정리. `SUID_Counter`의 잔존 `detected`/`changed` 필드는 익명 nested `struct var{}`로 묶어 의미 분리(카운터 vs slist 임시 단발 플래그). 이번 주 5커밋, 진행 중. 안승민과 송신 메시지 스키마 합의·Safe 모드 정책만 남은 단계.

**Agent Server 접속 시나리오 통합 + 외부 단말 자가 복구 경로 복원 (OR-1449)**. 한 단말이 매 부팅마다 사내망 사설 IP(`192.168.100.211:3181`) 하나에만 등록 시도하다 매번 실패하는 상태가 자가 복구되지 않던 사고를 진단·해결. 본사 서버 배포 도중 응답의 `doc.Servers`가 잠깐 1개(사내망)로 줄어든 순간을 받은 외부 단말이, 받은 1개로만 `RegisterToServer2`를 시도하다 응답이 안 오니 새 서버 목록도 못 받고, OR-1274에서 자동 시도 회복을 일부러 끊으면서 단말 스스로 고칠 길도 같이 막혀 있던 케이스다. 사내 단말은 사설 IP에 닿을 수 있어 문제가 안 드러나고, 연구소나 외부망 단말만 영원히 닿지 못한다.

접속 우선순위를 `Override Server → Servers → LastServer → DefaultServer`로 명문화하고 `LastServer` 개념을 추가했다. `LastServer`는 그 단말이 한 번이라도 등록 성공한 서버다. 한 번이라도 성공 이력이 있는 단말은 다음 부팅부터 `LastServer`로 돌아가 자기 자리를 다시 찾는다. `LastServer`가 채워져 있으면 `DefaultServer`는 시도 후보에서 영원히 빠진다 — 고객사 단말이 우리 회사 기본 매니저로 가버리는 사고를 동시에 막는 정책이다. UI 저장 분기의 "기존값과 같으면 무시"(`command.cpp:262`) 가드도 제거 — 사용자가 같은 값을 다시 저장해도 무조건 `SetServerOverride`를 호출하므로 다음 등록 사이클에서 그 주소로 다시 시도되고, 서버가 정상 응답으로 새 서버 목록을 내려주면서 단말이 자연스럽게 정상화된다.

변경 이력 추적을 위해 `server.log` 폭주 정리도 같이 했다. `module/logger.h`에 `dedup_sink`를 추가 — 같은 줄이 계속 들어오면 메모리에서 횟수만 세고, 다른 줄이 들어오는 순간 `[REPEAT N]` 한 줄을 자동 출력한다. 호출하는 쪽 코드는 손대지 않는다. `RegisterToServer2` 출력을 한 사이클당 한 줄로 정리한 뒤, 35회 반복이 2줄로 압축되는 것을 빌드 검증으로 확인. 4MB 한도는 그대로 두되 폭주가 사라져 한도에 닿는 시간이 훨씬 길어졌다. 다음 단계는 단말마다 서버 주소가 언제 어떻게 바뀌었는지의 영구 기록(`node_server_history` 컬렉션 + MQTT 전송)과 매니저에서 단말 서버 주소를 원격으로 정정하는 명령. 이번 주 1커밋, 완료.

## smahn9123 — REST API · MQTT · MongoDB 다섯 축 동시 정리

**프로세스 영향도 위젯 — 서버 구현 4-레포 동시 정리 + subscriber-driven publish (OR-1442)**. OR-1334 분석가 가이드를 서버 입장에서 검토·확장해 rest-api · service · migrations · package 네 서브모듈에 걸쳐 동시 정리. 지난 주에 굵직한 설계 결정을 끝낸 뒤 이번 주에는 e2e 통합과 후속 follow-up을 마무리해 In QA로 진입했다.

핵심은 subscriber-driven publish 모델이다. 개발 서버 기준 `sprocess` 컬렉션 SUID는 31,117개. 이 규모를 stateless로 전수 publish하면 시간당 ~224k msg가 되는데, 활성 매니저가 보는 SUID는 1개 ~ 수십 개이므로 99% 이상이 아무도 안 듣는 publish다. manager-web이 관심 SUID를 `PUT /v3/sprocess/subscribe`로 등록하면 Redis ZSET에 `ZADD`(`score=expire epoch sec`)로 추적하고, aggregator는 publish 직전 `ZRANGEBYSCORE sprocess:subscribers:{bucket} now +inf`로 활성 SUID set을 조회해 집계 결과와 교집합만 publish한다. `body.suids`는 그 시점 manager의 관심 SUID set(전체 snapshot)이다. keepalive: manager-web이 약 3분마다 같은 PUT을 재호출 → score 갱신. 다수 manager가 같은 SUID를 공유하면 각자 keepalive가 독립 적용되어, 어느 한 명의 위젯 종료가 다른 manager의 publish에 영향을 주지 않는다. 명시적 unsubscribe는 없고 ZADD-only 시맨틱 — score `< now`로 자연 만료될 때만 사라진다. cleanup은 PUT 호출마다 `ZREMRANGEBYSCORE -inf now`를 L-pipeline 1 RTT로 동반. abuse 가드: ZADD 후 `ZCARD` 검사로 한 bucket의 ZSET 멤버 수가 `MAX_ZSET_SIZE = 100,000`을 넘으면 방금 추가한 SUID들을 `ZREM`으로 롤백 + 429. 정상 운영(매니저 수 × 위젯 × SUID 수십개 ≈ ~100)이 cap의 1000배 여유 안에 있어 일반 사용엔 절대 닿지 않는다.

이번 주 마무리 작업은 `aggregated/list`의 SUID 메타 8 필드 합류(B-7)다. 분석가 follow-up — "SUID만 보면 관리자가 어떤 실행파일인지 알기 어렵다. `ProcName / ProductName / ProductVersion / CompanyName / FileName / FileVersion / FileDescription / MD5`가 있으면 좋겠다". 이 8 필드는 SUID 단위로 고정(`PRunUID+FastHash+RunUID` hash이므로 한 SUID = 한 실행파일)이고, `sprocess` raw의 71.4%가 `ProductName` 보유, 8.8%는 모든 doc에서 메타 0%, 19.6%는 doc별 혼재. `GetSprocessAggregatedListCase`의 pipeline 끝에 `$skip/$limit` 뒤에서 `$lookup sprocess`로 채택 — `sprocess_hourly` schema 무변경, aggregator 무변경, 변경 폭 최소. 같은 SUID의 메타 보유 doc을 우선 take하기 위해 sub-pipeline `$sort {ProductName: -1, _id: 1} + $limit 1`로 mongo null sort 규칙을 활용한다. `sprocess {SUID:1}` 인덱스가 필수(기존 복합 인덱스 prefix=`id`라 SUID-only COLLSCAN 4.7s) → v0013 마이그레이션에 추가. 별도 systemd 서비스 `orange.process-aggregator.service`로 격리, hourly cron `:02` UTC + partial `:05/:15/:25/:35/:45/:55` UTC, daily cron `00:30` 서버 로컬 + partial `06:15 / 12:15 / 18:15` 서버 로컬. 위젯이 새 hour 시작 후 최대 5분 안에 진행 중 hour 데이터를 본다. 이번 주 6커밋, In QA.

**MQTT broker 드롭/백프레셔 위젯 노출 + WS 모드 크래시 수정 (OR-1412)**. Mosquitto가 publish하는 메시지 드롭/백프레셔 관련 `$SYS` 토픽 4개를 수집·노출해 슬로우 컨슈머나 트래픽 burst로 인한 broker QoS 0/1/2 메시지 손실을 운영자가 실시간 감지할 수 있게 만들었다. `$SYS/broker/publish/messages/dropped`(누적 드롭), `$SYS/broker/load/publish/dropped/1min`(분당 드롭율 이동평균), `$SYS/broker/store/messages/count`(스토어 보관 — 적체 지표), `$SYS/broker/clients/disconnected`(영속 세션 보유 오프라인 클라이언트). 설계 단계에서 4번째 지표로 잡았던 인플라이트(`$SYS/broker/messages/inflight`)는 운영/dev의 mosquitto 2.0.18에 해당 토픽이 존재하지 않아, 동일하게 백프레셔/적체를 가늠할 수 있는 스토어 보관(`$SYS/broker/store/messages/count`)으로 대체. drop·drop율 > 0이면 `text-red-500`으로 강조. 매니저 위젯에서 `wsConnected` 식별자가 미정의되어 WS 모드(`useWebSocket: true`)에서 `ReferenceError`로 렌더가 깨지던 런타임 크래시도 같이 수정 — 헤더의 파란 "Live" 뱃지가 정상 표시되도록 했다. Playwright MCP로 dev 라이브 데이터에 OrangeServerTab을 레이어 윈도우 팝업으로 띄워 WS/폴링 두 모드 모두 검증. 이번 주 완료.

**REST API 3181 nginx 우회 스캔 fail2ban HTTP jail + IP 위조 방어 (OR-1448)**. REST API 컨테이너 포트 3181은 Agent 직접 접속용으로 인터넷에 공개되어 있어, nginx(443)를 우회한 외부 스캐너가 `GET /`·`/favicon.ico`·`/.well-known/security.txt` 등으로 404 폭주를 일으킨다. compose 로그를 journald + `logging.options.tag`로 `CONTAINER_TAG=orange-rest-api`로 고정해 컨테이너 이름이 환경마다 달라도(고객사 `orange-rest-api` / dev `rest-api`) 한 벌의 jail이 모든 환경에서 동작하게 했다. `banaction_allports`로 시스템 방화벽 백엔드(nftables/iptables)를 자동 추종해 전 포트 차단, `maxretry=10`·`bantime/findtime/ignoreip/bantime.increment`는 `[DEFAULT]` 상속, recidive 메타 jail 연동. `/api` 404는 `ignoreregex`로 분리해 정상 요청을 살림. 짝지은 OR-1156은 client IP 위조 방어 — `get_client_ip` 신설(`app/utils/client_ip.py`). TCP peer가 공인이면(직접 연결) forwarded 헤더를 무시·peer를 사용, 사설/루프백(nginx 경유)이면 `X-Real-IP` 신뢰. 이로써 외부 스캐너가 XFF 위조로 ban 회피·무고한 IP 오폭하는 것을 차단. dev·test 서버 6/6 시나리오 PASS. 이번 주 완료.

**Rule을 Git 서브모듈(Orange-Rules)로 분리 (OR-1457)**. 기존에 agent 레포 하위 폴더로 존재하던 rule을 독립 Git 서브모듈로 완전히 분리. 그동안 rule 관리자(데이터 분석자)가 로컬에서 관리하던 "최신 안정" rule을 package 레포의 `rule/` 폴더에 수동 반영해 왔고, makepackage가 이를 제품 패키지에 복사했다. 이제 서브모듈의 main 브랜치가 항상 안정판을 보장하므로, package 레포는 더 이상 rule 사본을 보유하지 않는다. `package/rule/` 폴더 전체 삭제(268개 파일), `makepackage.sh`의 rsync 소스를 `$BASEDIR/rule/` → `$BASEDIR/../rules/`(루트 서브모듈)로 변경, `makepackage.bat`의 RoboCopy 소스도 `%BASEDIR%rule` → `%BASEDIR%..\rules`로 변경(CP949 + CRLF 인코딩 유지). 패키지 대상 경로(`{VER}/@update/rule/`)는 변경 없음 — 에이전트가 기대하는 패키지 레이아웃 유지, 빌드 산출물 구조 동일. 빌드 전 동기화는 서브모듈별 `git pull` 또는 `git submodule update --remote` 방식이어야 하며, `git submodule update`만 실행하면 핀된 옛 커밋으로 되돌아간다(주의). 이번 주 완료.

원인: 현대 Windows 11은 누적 업데이트(LCU)를 CBS(Component-Based Servicing) 경로로 설치한다. 이 경로는 EventID 19를 남기지 않고 `Setup` 채널 / `Microsoft-Windows-Servicing` Provider / EventID 2를 남기며, 설치 주체는 `Client = UpdateAgentLCU`다. `Get-HotFix` 교차 검증으로 실제 설치가 EventID 2로 기록됨을 확인. 신규 트리거 XML `BinSnap-OnServicing.xml`을 추가하고 성공한 설치만 거르도록 UserData 조건까지 XPath에 포함: `*[System[Provider[@Name='Microsoft-Windows-Servicing'] and (EventID=2)]] and *[UserData/CbsPackageChangeState[IntendedPackageStateTextized='Installed' and ErrorCode='0x0']]`. 결과는 트리거 3개 체계 — 정기 백업(매주 일요일 03:00), 레거시 WU(EventID 19, 보험), 현대 CBS/LCU(EventID 2, 이 머신 실제 경로). 두 이벤트 트리거 모두 SYSTEM/Highest, Delay PT5M, IgnoreNew — 한 번의 업데이트로 EventID 2가 여러 번 발생해도 1회만 실행. 환경 무관 커버. 교훈: 문서(MS Learn)만 봤으면 "EventID 19 사용"으로 끝나 이 머신에서 동작하지 않았을 것이다. 실제 머신의 이벤트 로그를 직접 조회/교차검증한 덕에 CBS 경로(EventID 2)를 발견했다.

부가로 UpdateInfo 모듈을 추가해 "어떤 업데이트(KB) 때문에 시스템 프로세스가 추가됐는지" 인과를 diff에 함께 기록. EventID 19의 KB는 EventData의 `Data Name="updateTitle"`, EventID 2의 KB는 UserData의 `PackageIdentifier` — 구조가 달라 두 채널 동시 조회. `--list-updates=10080` 헬퍼(스캔/diff 안 거치고 KB 조회만 단독 실행, 7일치). EvtQuery 15001(`ERROR_EVT_CHANNEL_NOT_FOUND`) 이슈는 simple XPath의 `Provider[@Name='Microsoft-Windows-Servicing']` 필터를 EvtQuery가 채널로 해석하려다 `Path` 인자("Setup")와 충돌하는 문제로, structured query(트리거 XML과 동일한 `<QueryList><Query Path=...>` 형식)로 우회. `EvtQuery(NULL, NULL, structuredQuery, EvtQueryChannelPath | EvtQueryReverseDirection)`. 이번 주 완료.

## kimyh-orange — 분석 R&R 본업

**프로세스 영향도 맵 위젯 mockup_v6 매니저 위젯 포팅 (OR-1293)**. 데이터 분석에서 검증한 프로세스 영향도 맵 mockup_v6 디자인을 매니저 위젯으로 포팅. 본 티켓의 원래 요청은 "system 컬렉션 시간당 1건 강제 수집" 강화였으나, 이후 데이터 분석에서 발상 자체가 잘못됐음이 확인되어 폐기 처리되었다(`Won't Do`).

폐기 이유 — `system` 컬렉션은 (PC × 5분) 의 OHLC + 누적 통계 + 그 5분간 이벤트 카운트로, 본질이 "5분 단위 압축"이다. "시간당 1건 강제"는 5분 단위 의미를 파괴하는 발상이다. 그리고 `sprocess`가 이미 PC × 프로세스(SUID) 단위 시간당 1건 시계열을 추적 중이다 — 한 (PC, SUID)의 측정 분포 검증에서 174일 range에서 2,700 unique 5분 slot, `docs/hour = 1.0`. 즉 PC가 깨어있는 동안 한 SUID = 시간당 1건 정확히 측정. system 강화 안 해도 sprocess로 시계열 baseline 산출 가능하다. 그리고 진짜 부족함은 데이터 양이 아니라 PC 가동 시간이다 — system이 sparse한 진짜 원인은 PC가 켜져 있는 시간이 적은 것(출근 8h / 퇴근 16h)으로, Agent 측 수집 강화로는 해결이 안 된다.

A+D prototype 검증 결과로 폐기를 뒷받침했다. chrome.exe 도입 ±14일·임계값 10건·`sprocess+system` 5분 슬롯 합집합 기준에서, system 단독(±7일·≥50건)이면 가능 PC 5개(11%)였던 것이 A 조합(±14일·≥10 슬롯·union)이면 21개(46%)로 4배 확장. 시스템 강화 없이 임계값 + 윈도우 + 합집합 조정만으로 가능 PC가 4배 확장된다는 결론. 신뢰도 100% 결론을 위한 진짜 데이터 갭(★★★ PC 가동/슬립 이력 명시 — Agent boot/shutdown 이벤트, ★★★ 사용자 활성 시간 — 키보드/마우스 input·포어그라운드 윈도우, ★★★ 변경 이벤트 통합 타임라인 — filelist + nodeinfo 변동 + OS 패치, ★★ 애플리케이션 그룹핑 — chrome 계열 매핑, ★★ 사용자 변경 이벤트)도 별도 ticket 후보로 분리해 정리. 이번 주 1커밋(매니저 위젯 포팅).
