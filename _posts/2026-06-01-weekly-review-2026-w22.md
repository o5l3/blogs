## smahn9123 — 서버 백엔드 전방위 진전

**프로세스 영향도 위젯 — 서버 구현 4-레포 동시 정리 (OR-1442)**. OR-1334 분석가 가이드(V5.15)를 서버 입장에서 검토·확장해 rest-api · service · migrations · package 네 서브모듈에 동시 머지. system aggregation의 평균값 소스가 OHLC `close`(통상 마지막 raw)와 어긋나는 점을 잡아 `average` 필드를 OHLC와 별도로 신설했고(MIN_COUNT_FOR_OHLC=5 미달이어도 첫 sample부터 sum/count로 채움), daily 경계는 서버 로컬 timezone 자정으로 통일하되 DB 저장 값은 모두 UTC로 일관화. `aggregated/list`는 같은 SUID가 여러 hour|day에 등장할 때 row가 중복되던 P-1을 버리고, aggregator가 이미 보관해 둔 `*_sum` / `CounterCount_sum`으로 SUID 단위 가중평균(`cpu_percentage = ProcessTime_total / SystemTime_total * 100`)을 즉시 산출하는 P-2로 재설계했다. fast path 자동 흡수 — 시간 범위 안에 bucket이 1개뿐이면 `$group`도 doc 1개만 통과 → 단일 pipeline로 최적.

핵심은 subscriber-driven publish 모델이다. 31k SUID 전수 publish는 시간당 ~224k msg로 99%+가 아무도 안 듣는 publish이므로, manager-web이 관심 SUID를 `PUT /v3/sprocess/subscribe`로 등록하면 Redis ZSET에 `ZADD-only`(3분 keepalive·5분 TTL score 자연 만료)로 추적하고 aggregator는 `ZRANGEBYSCORE`로 활성 SUID set만 publish한다. abuse 가드로 한 bucket의 ZSET member 수가 `MAX_ZSET_SIZE = 100,000`을 넘으면 방금 추가한 SUID들을 `ZREM`으로 롤백 + 429. 정상 운영(매니저 수 × 위젯 × SUID 수십개 ≈ ~100)이 cap의 1000배 여유 안에 있다. 별도 systemd 서비스(`orange.process-aggregator.service`)로 격리, hourly cron `:02` + partial `:05/:15/:25/:35/:45/:55`로 위젯이 새 hour 시작 후 최대 5분 안에 진행 중 hour를 볼 수 있게 했다. 이번 주 39커밋의 최대 작업.

**조직도 일괄 로드 단일 트랜잭션화 + Beanie `bulk_writer` 단일 RTT (OR-1435)**. `POST /v3/organization/load`의 `clear → load_groups → load_users` 사이 중간 실패에 따른 부분 적재 노출을 단일 MongoDB 트랜잭션으로 묶었다(전제: 아래 OR-1439). `load_groups` · `load_users` · `recompute_all_node_user_fields`를 Beanie `bulk_writer`로 전환해 RTT를 1회로 줄였고, `User.group_code` 사전 검증을 엄격 모드로 추가, 조직도 로드 후 ghost 그룹의 활성 Command 자동 정리·Node 비정규화 필드 일괄 재계산·고아 parent 처리 옵션화(`strict_orphan_parents`)까지 시나리오 6번을 모두 정리했다. 5번 시나리오(코드 관성)는 Agent 측 동작 실측으로 무영향임을 확인하고 별도 리팩터링 이슈로 분리.

## kim-tigerj — Manager·Agent 회귀 추적과 BSOD 분석

**nodeInfo will aux abrupt close 우회 — 크롬 148 + OpenSSL 3.0.x 잔여 경로 회피 (OR-1418)**. OR-1409로 위젯 MQTT를 글로벌 일원화한 직후 `nodeInfo cleanup` 일부가 빠져 위젯이 닫혀도 Agent가 "이 화면 더 안 본다" 신호를 받지 못해 전용 패킷이 누적되던 회귀. broker 디버그 로그(`log_type all` + `connection_messages true`) · tcpdump · 엣지 vs 크롬 격리 비교로 크롬 148의 WebSocket close 시퀀스가 broker OpenSSL 3.0.x record-layer 버그의 `error:0A000123:SSL routines::application data after close notify` 잔여 경로를 다시 트리거하고, 그 결과 무관한 Agent 연결이 끊긴다는 사실을 확정. OR-1414 본문이 "운영 로그상 0건이라 100% 커버"라고 한 잔여 경로가 크롬 148 업데이트로 0건이 아니게 된 것이다. `SSL_OP_IGNORE_UNEXPECTED_EOF`는 `0126` 전용이고 0123을 무시할 옵션은 OpenSSL 3.0.13 헤더에 존재하지 않아 shim 확장도 불가능.

해결: will 보조 연결을 `abrupt close (end(true) = stream.destroy())`로 전환. graceful close가 0123을 트리거하는 것이 핵심이므로 abrupt는 `close_notify`를 생략해 0123을 원천 발생 불가로 만들고, 대신 발생하는 `0A000126:unexpected eof`는 OR-1414 shim이 흡수한다. 동시에 abrupt close가 broker의 WILL을 즉시(~1ms) 발사시켜 통지 안전망이 살아남는다. 매니저 측 두 곳에 상세한 `[중요]` 경고 주석과 비활성 publish 백업 코드를 박아 "graceful이 정상적인 종료"라는 단순 웹개발 상식으로 되돌리면 0123 재발 + WILL 미발사 두 문제가 컴파일·런타임 에러 없이 조용히 동시 발생하는 함정임을 박제했다. 실측 — `nodeInfo` 10회+ 반복 여닫이에 0A000123 6건 → 0건, agent Protocol error drop 발생 → 0건, broker 소켓 CLOSE-WAIT 누적 0건. 근본 처방(broker OpenSSL 3.2+ 또는 1.1.1 교체)은 별도 진행.

**Manager > 노드 통계 WebSocket 야간 유실 — close 코드별 정책 정립 + 인스턴스 가드 (OR-1129)**. OR-1105 상태 패널로 적발된 야간 노드 통계 WebSocket(`/api/v3/ws/node/stats`) 단일화 현상에 대해, `wasClean === true`라도 close 코드는 정상이 아닐 수 있다(서버 `WS_1011` 내부 에러도 핸드셰이크가 정상 완료되면 `wasClean = true`)는 점을 짚고, 클라이언트가 close 코드별로 자동 복구하도록 정책을 정립. 1006(네트워크 끊김) / 1011(서버 내부 오류)은 지수 백오프 재연결(`1s → 2s → … → 30s`), 1000(정상) / 1005(코드 없음)은 무동작, 그 외 + 1008(인증/정책 위반)은 강제 로그아웃. 정상 메시지 수신 시 백오프 카운트 리셋으로 "수락 직후 끊김"이 1초마다 반복되는 플래핑을 차단하고, 인스턴스 가드("지금도 내가 현재 소켓일 때만" 동작)로 늦게 도착한 옛 소켓 이벤트가 새 연결을 덮어쓰는 것까지 막았다. 로그아웃 시 `useNodeStatsStore.getState().reset()`으로 옛 토큰 좀비 재연결도 차단. "메인 MQTT는 끊기면 무조건 로그아웃, node-stats만 degraded 복구"라는 운영 원칙도 명시화.

**[Manager] ESLint `no-shadow` 규칙 + git pre-commit 게이트 — OR-1028 11일 잠복 버그 재발 차단 (OR-1029)**. OR-1028 파일 다운로드 버그가 `import { downloadFile }`과 컴포넌트 내부 `const downloadFile = useCallback(...)`의 변수 섀도잉으로 11일간(03-17~03-28) 잠복했던 사고를 받아, 새로 추가·수정하는 `.ts`/`.tsx`에 `no-shadow` 위반이 있으면 커밋이 거부되는 pre-commit 훅을 도입. `yarn install` 한 번에 윈도·맥·WSL 공통 자동 설치(설치 스크립트는 Node로 동작하므로 bash 불필요), Node가 없으면 통과가 아니라 차단으로 통일. 의도적 섀도잉만 인라인 `// eslint-disable-next-line @typescript-eslint/no-shadow` 주석으로 허용. 같은 사고를 도구 차원에서 들어오지 못하게 막은 마무리.

**MSI 노트북 BSOD #2 — vmmemWSL 어서션 + e1dn.sys D3 전환 두 케이스 windbg 분석 (OR-1307)**. WSL2 `vmmemWSL`이 일으킨 `0x7E SYSTEM_THREAD_EXCEPTION_NOT_HANDLED`(`STATUS_ASSERTION_FAILURE 0xC0000420` in `nt!VmpInvalidateSingleGpaRange+0x44`)는 Windows 11 24H2 26100.8246의 WSL2 메모리 관리 회귀로 추정. 별도 발생한 `0x9F DRIVER_POWER_STATE_FAILURE`는 windbg `!irp`로 `IRP_MJ_POWER → SET_POWER D3`가 `\Driver\e1dnexpress`에 도달했다가 시한 내 완료되지 못한 케이스로 확정. 충돌 위치 `NxNblDatapath::SetRxHandler+30`(`nxnbldatapath.cpp:92`)에서 NetAdapterCx의 RX 핸들러 분리 단계(`SetRxHandler → rundown wait`)에서 멈춘 것. 2024년 9월 빌드 `e1dn.sys`(Intel `PCI\VEN_8086&DEV_550F` 이더넷 컨트롤러)는 그동안 Intel·MS가 NetAdapterCx D3 전환 패치를 여러 차례 배포한 상태라 벤더 본 드라이버 업데이트와 어댑터 전원 관리 옵션 해제로 회피하도록 조치 정리.

**sprocess 1시간 단위 hourly 집계 인프라 — 부팅 누적 → 시간 집계 의미 교체 (OR-1340)**. OR-1306이 깐 1시간 주기 송신 인프라 위에서 송신 수치를 부팅 누적이 아니라 1시간 단위 집계로 바꾸는 작업. `CSProcessHourly` 클래스(`yagent21/CSProcessHourly.h`)에 SUID 기반 메모리 버킷 + `Rotate` baseline carry(`new.start2 = MAX(old(start2+start-stop), 0)`)·`Cleanup` 2시간 비활성 evict·`DebugFlush` dev 모드를 한 클래스로 모으고, 저장 `Running` 컬럼을 sprocess/sprocess_all에서 제거해 8개 SELECT에서 `MAX(Start-Stop, 0) AS Running` 파생으로 전환(저장 컬럼 없음 + UPDATE 쿼리 없음 + 0복귀 자동). `wal_autocheckpoint` 노브를 db/summary.json 헤더에 신설해 외부 reader(`orangedb.exe`)에서 라이브 데이터 즉시 visible. 28시간 무중단 운영에서 40회 hour rollover carry mismatch 0건, msedge SUID 3081108326 02:00 Running=4 → 03:00 Start2=4 carry 일치 검증. SystemStat MQTT 송신 빈도 폭증도 평상시 60초 / LiveResponse 5초로 같이 정리. 송신 메시지 스키마 합의(`smahn9123`)와 Safe 모드 정책만 남김.

**Agent `doc.Servers` 잔재로 인한 접속 불가 — `dedup_sink` + UI 저장 무효 가드 제거 (OR-1449)**. ## wychoi-orangelabs — 커널 드라이버 파일 이벤트 throttle 본격 적용

**파일 삭제·오류 필터링 — 시간 기반 throttle + 512-entry LRU 테이블 (OR-1427)**. `File.Failed.cpp` / 삭제 이벤트의 driver→agent notify 횟수가 한 달 동안 17조 회 누적 관측됨(분당 약 39만 건 통과로 agent 큐 폭주 위험). 기존 정책 `count % 100 == 1`은 동일 `(process+status+file)` key 마다 매 100회 1회 통과라 한 key가 지속 폭주하면 100건당 1건씩 무한 통과되어 폭주 자체는 막지 못하는 한계가 있었다. 두 가지를 동시에 바꿨다.

첫째, 횟수 단위 → 시간 단위. `FILTER_COUNT_NOTIFY_INTERVAL (100)` → `FILTER_COUNT_NOTIFY_INTERVAL_SEC (600)`로 10분당 1회 전환. 시간 source는 `KeQueryUnbiasedInterruptTime`(100ns 단위, sleep 시간 제외)을 써서 노트북이 절전 상태였던 시간은 카운트하지 않게 했다 — 깨어난 직후 즉시 notify가 폭주하는 패턴을 차단.

둘째, 카운트 테이블을 LRU로. 기존 hash table(open addressing) + `ERESOURCE`는 포화 시 `RtlZeroMemory`로 전체 reset하여 모든 key의 `LastNotify` 손실 → 다음 발생들이 "첫 발생"으로 인식되어 즉시 notify 통과 → throttle 무력화되던 문제와, `ERESOURCE`(56B)가 read/write 구분 없는 use case에 과한 도구라는 문제를 동시에 안고 있었다. 신규 자료구조는 512-entry linear array(`FILTER_COUNT_SLOT[512]`) + `EX_PUSH_LOCK`(8B, 1/7 크기, contention 무 시 atomic CAS만). 포화 시는 가장 오래된 `LastNotify` entry 1개만 LRU evict해서 다른 entry의 `LastNotify`를 보존 — throttle 효과가 살아남는다. `linear scan` 한 번에 기존 key 검색 + oldest 추적 동시 처리.

API도 통합. `FilterCountIncrement` → `FilterCountIncrementAndCheckInterval(Key, IntervalSeconds, &Count)` 로 바뀌어 `TRUE` 반환 시 호출자가 `FfNotifyFailed` 호출, `FALSE`면 throttle 차단. 호출 site의 `count % N == 1` 분기가 사라져 코드 단순화. 효과 비교 — 17조 회 케이스 1,700억 건 notify → 4,320건(약 4천만 배 절감), Lock 크기 56B → 8B, Table 메모리 2048 × 24B = ~48KB → 512 × 24B + 16B = ~12KB.

PE Authenticode 파싱 비용을 회피하는 `SIGNER_CACHE`(`ofilter/File.Signature.h/cpp`)도 같이 들어가, ProcessNotify에서 `ResolveSignatureLevelCached(pProc, PID, pPathHint) → pPsGetProcessSignatureLevel → SL≥7이면 ProcPath 키로 캐시 등록` 흐름으로 매 파일 이벤트마다 디스크 I/O 3회 + PagedPool 최대 16MB이 들어가던 PE 파싱 후보 설계(OR-1423)를 spinlock + atomic CAS 수 마이크로초로 대체. `WindowsSystemBinary` 화이트리스트도 단일 `g_SystemAppsPrefix`에서 8개 prefix 배열(SystemApps · WindowsApps · OneDrive · Microsoft x86 · Office 64/32bit · Common Files microsoft shared · Defender Platform) + Office/OneDrive/MsMpEng/WmiPrvSE 등 basename 30여 개로 확장.

**Agent > orange.exe > MS 시스템 프로세스 `bSystem` TRUE 설정 (OR-1434)**. `PROCESS_ENTRY.IsSystemProcess`(`ofilter.h:242`)에 `WsbClassifyBinary` 결과가 정상 저장되는데도 `Y_PROCESS_MESSAGE.bSystem`(`yagent.define.h:915`)이 false 하드코딩이라 svchost · explorer 같은 일반 시스템 프로세스가 PPL 판정 / 파일 속성 판정 단계 전까지 `bSystem = false`로 전달되던 정합 오류를 수정. ofilter의 `CreateProcessMessage` 오버로드 1·2 + `Process.cpp:422` `ProcessStart` 콜백 + yagent21 `CProcess2.cpp:291` 7개 사이트를 손봐 드라이버 경로 기반 판정 결과가 `AddProcessCache` 진입 시점부터 즉시 Current DB `process` 테이블 · Summary DB `filelist` 테이블 · MQTT `public/node/filelist` 토픽까지 전파되도록 일원화. 3단계 false → true 단방향 승격 구조(① 드라이버 `WsbClassifyBinary` ② PPL ③ `SetSystemFileFlag2` 파일 속성)는 그대로 유지.
