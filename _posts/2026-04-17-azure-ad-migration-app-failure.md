## 개요

전사 AD 도입과 함께 한 고객사가 Microsoft Azure AD를 처음 도입했습니다. (고객사·담당자명 비공개)

- 사번·회사명이 포함된 새 AD 계정으로 Windows 로그인 체계 전환
- 기존 로컬 계정 정보는 전환 솔루션으로 새 AD 계정에 이관
- 이후 일부 PC에서 **기존 계정으로 설치한 프로그램의 장애**(화면 깜박임, Excel 동작 불량 등) 발생

## 문제의 원리

핵심은 **기존 계정 권한으로 설치된 프로그램을 새 AD 계정 권한으로 실행**하는 과정에서 발생합니다.

### 프로그램의 설치·동작 방식

**파일**
- `C:\Windows`·`C:\Program Files`·`C:\ProgramData` 같은 **공용 디렉터리**
- `C:\Users\[계정]\AppData\Local` 같은 **계정별 개별 디렉터리** — 생성한 계정만 접근 가능하도록 권한 설정
- 프로그램에 따라 둘 다 사용

**레지스트리**
- `HKEY_LOCAL_MACHINE\SOFTWARE` 공용 / `HKEY_USERS\[계정SID]` 개별
- 프로그램이 사용하는 레지스트리 공간에 **파일 경로 정보**를 저장하기도 함

### 두 가지 대표 오류

**ERROR_FILE_NOT_FOUND (Code 2)**
- 프로그램이 공용이 아닌 **기존 계정의 디렉터리·레지스트리**에 데이터를 기록한 경우
- 새 계정으로 실행하면 새 계정 공간에서 데이터를 읽으려 하나 존재하지 않음
- 기존 계정 디렉터리의 바이너리(PE)를 동적 로드하려는데 새 계정 디렉터리엔 없음

**ERROR_ACCESS_DENIED (Code 5)**
- 새 계정으로 실행되는 프로그램이 **기존 계정 디렉터리 경로를 내부 참조**
- 파일이 기존 계정 디렉터리에 존재해도, 새 계정은 권한이 없어 접근 불가

## 왜 전환이 어려운가

전환의 핵심은 위 두 오류를 막는 것이지만, 프로그램마다 디렉터리·레지스트리 위치와 내부 동작이 달라 모든 경우를 다루기 어렵습니다.

- 파일은 기존→새 계정 디렉터리로 복사 가능
- **레지스트리는 계정별 고유 공간이라 복사가 어려움**
- 복사하더라도 레지스트리 값에 `C:\Users\기존계정\...` 경로가 박혀 있으면, 새 계정 프로그램이 그 값을 읽어 **여전히 기존 계정 파일을 참조**
- 일부 프로그램은 경로를 **자체 암복호화**로 저장해 변환 불가
- 계정 비종속 프로그램은 공용 공간을 쓰지만, **계정 종속 프로그램**은 위치가 계정 따라 달라져 문제 발생

## 임시 해결

문제가 되는 프로그램을 **새 계정으로 로그인한 상태에서 삭제 후 재설치**합니다. 사용 디렉터리·레지스트리가 새 계정 권한으로 올바르게 재구성됩니다. 근본 해결을 위해서는 장애 프로그램 목록을 전환 솔루션 업체에 전달해 케이스별 대응이 필요합니다.
