## smahn9123 — 서버 백엔드·인프라 전방위

5월 한 달 178커밋, Jira 40건 완료 — 기간 최다. 신규 큰 작업 세 개를 동시에 굴리면서 보안·인프라 마무리까지 같이 묶었다.

**프로세스 영향도 위젯 — 서버 구현 4-레포 동시 정리 (OR-1442)**. OR-1334 분석가 가이드(V5.15)를 서버 입장에서 검토·확장해 rest-api · service · migrations · package 네 서브모듈에 동시 머지. system aggregation의 평균값 소스가 OHLC `close`(통상 마지막 raw)와 어긋나는 점을 잡아 `average` 필드를 OHLC와 별도로 신설했고(MIN_COUNT_FOR_OHLC=5 미달이어도 첫 sample부터 sum/count로 채움), daily 경계는 서버 로컬 timezone 자정으로 통일하되 DB 저장 값은 모두 UTC로 일관화. `aggregated/list`는 같은 SUID가 여러 hour|day에 등장할 때 row가 중복되던 P-1을 버리고, aggregator가 이미 보관해 둔 `*_sum` / `CounterCount_sum`으로 SUID 단위 가중평균(`cpu_percentage = ProcessTime_total / SystemTime_total * 100`)을 즉시 산출하는 P-2로 재설계.

핵심은 subscriber-driven publish 모델이다. 31k SUID 전수 publish는 시간당 ~224k msg로 99%+가 아무도 안 듣는 publish이므로, manager-web이 관심 SUID를 `PUT /v3/sprocess/subscribe`로 등록하면 Redis ZSET에 `ZADD-only`(3분 keepalive · 5분 TTL score 자연 만료)로 추적하고 aggregator는 `ZRANGEBYSCORE`로 활성 SUID set만 publish한다. abuse 가드로 한 bucket의 ZSET member 수가 `MAX_ZSET_SIZE = 100,000`을 넘으면 방금 추가한 SUID들을 `ZREM`으로 롤백 + 429. 별도 systemd 서비스(`orange.process-aggregator.service`)로 격리, hourly cron `:02` + partial `:05/:15/:25/:35/:45/:55`로 위젯이 새 hour 시작 후 최대 5분 안에 진행 중 hour를 본다. 5월 한 달 39커밋의 최대 작업.

**조직도 일괄 로드 단일 트랜잭션화 + Beanie `bulk_writer` 단일 RTT (OR-1435)**. `POST /v3/organization/load`의 `clear → load_groups → load_users` 사이 중간 실패에 따른 부분 적재 노출을 단일 MongoDB 트랜잭션으로 묶었다(전제: 아래 OR-1439). `load_groups` · `load_users` · `recompute_all_node_user_fields`를 Beanie `bulk_writer`로 전환해 RTT를 1회로 줄였고, `User.group_code` 사전 검증을 엄격 모드로 추가. 조직도 로드 후 ghost 그룹 활성 Command 자동 정리·Node 비정규화 필드 일괄 재계산·고아 parent 처리 옵션화(`strict_orphan_parents`)까지 시나리오 6번을 모두 정리.

**MongoDB single-node replica set 전환 — multi-document 트랜잭션 인프라 (OR-1439)**. OR-1435 트랜잭션을 가능하게 하기 위해 standalone MongoDB를 single-node replica set으로 전환. `setup.sh` / `update.sh`에 `keyFile`(`openssl rand -base64 756`, `mongodb:mongodb`, 400) 생성과 `rs.initiate` 3단 분기(무인증 OK / `AlreadyInitialized` SKIP / `Unauthorized` → admin auth fallback)를 idempotent로 짜고, PyMongo의 `hello` 응답 자동 발견을 위해 멤버 host를 rest-api · service 양쪽이 도달 가능한 docker network gateway IP로 설정(orange_net 172.30.0.1 / default bridge 172.17.0.1). 테스트 · 개발 서버 모두 데이터 100% 보존 마이그레이션 검증, rest-api/service 코드 변경 0건.

**KISA SW 개발보안 가이드 — Phase 1~3 시큐어코딩 20건 마무리 (OR-994)**. KISA(2021) 제4장 구현단계를 기준으로 servers/rest-api와 servers/service에서 도출한 20건의 보안 결함을 일괄 처리하고 본 이슈 종료. CWE-798 마스터키·MD5·`deploy.sh` 평문 비밀번호, CWE-22 Path Traversal·Zip Slip(`extractall` 시 `../../` 차단), CWE-918 모니터 SSRF(loopback `127.0.0.0/8 ::1` · link-local `169.254.0.0/16 fe80::/10` 차단을 문자열 비교가 아닌 DNS 후 IP 레벨에서), CWE-307 로그인 IP Rate Limiting(`POST /v3/manager/login` 60초/20회 Redis pipeline `INCR+EXPIRE` 원자 처리), CWE-330 Node ticket UUID v1→v4, CWE-434 업로드 크기·Zip bomb 가드(`/v3/rule/upload` 10MB + 전개 200MB), CWE-312 `v1 logon` 응답에서 비밀번호 해시 제거까지. CORS 와일드카드와 `verify=False` 두 건은 사내 PKI 미비를 사유로 위협 모델 범위 밖으로 분리하면서도 향후 활성화를 위해 `# WARN: cert verification disabled — internal CA 도입 시 활성화` 주석을 7개 위치에 박아 두었다.

**fail2ban 통합 + REST API 3181 HTTP jail (OR-1313, OR-1448)**. 인터넷 노출 SSH 무차별 대입을 systemd journal 백엔드 + nftables banaction + 지수 증가 ban + recidive 메타-jail(24h 내 5번 ban → 1주 장기차단)로 차단. `firewall.rules`에 `__SSH_PORT__` 토큰을 도입해 install 시점에 `detect_ssh_port` 헬퍼가 동적 치환하도록 만들어 고객사마다 다른 sshd 포트를 자동 반영. Agent 직접 접속용으로 의도된 3181 노출에는 별도 HTTP jail을 추가 — compose 로그를 journald + `logging.options.tag`로 `CONTAINER_TAG=orange-rest-api`로 고정해 컨테이너 이름이 환경마다 달라도 한 벌의 jail이 동작하게 했고, `banaction_allports`로 시스템 방화벽 백엔드(nftables/iptables)를 자동 추종해 스캐너의 전 포트를 차단.

**broker 잔여 SSL 버그 우회 + 폐쇄망 동봉 (OR-1414)**. OpenSSL 3.0.x record layer 버그(partial TLS record + close 에러 시 SSL_CTX 공유 상태 손상)를 Ubuntu 24.04가 패치하지 않는 점을 격리 환경에서 확정한 뒤, `SSL_OP_IGNORE_UNEXPECTED_EOF` 플래그를 `LD_PRELOAD`로 주입하는 shim을 운영에 안착. 5월에는 폐쇄망 update 안정성을 위한 3단계 보강을 마무리했다 — ① silent skip 방지 안전망(install.sh 실패 시 setup/update가 die 대신 경고만), ② WSL Ubuntu 24.04(glibc 2.39)에서 빌드한 사전 빌드 `.so`를 git에 동봉 + Makefile `MAX_GLIBC=2.39` ceiling을 `make verify`가 강제, ③ Orange-Package CLAUDE.md에 트리·동봉 정책·setup/uninstall 단계 문서화.

**Service 계층의 API 스키마 의존 제거 — 순환 import 근본 정리 (OR-1425)**. Service / Infra가 API 스키마(`app/api/v3/schemas/*`)를 거꾸로 import하던 layer 위반 3건(CommandTemplate · Organization · infra/cgroup)을 모두 정상화. 각 계층 전용 `schemas/` 디렉터리에 입력/결과 모델 신설, `usecases/mapper/` 3개(CommandTemplate · Organization · Health)로 API↔내부 변환을 일관화. `__future__` + `TYPE_CHECKING` 회피책을 완전히 걷어내고도 외부 API 응답 변경 0건 — 순수 내부 리팩터링.

**nodedetect v3 마이그레이션 + 식별자·계층 정리 (OR-1424, OR-1355, OR-1294)**. `/v2/nodedetect/count(group_by_cuid_cgid=true)`를 `/v3/detect/nodes/count`로 옮기며 응답을 '고유 증상 인스턴스 수(distinctCount, 기존 v2 의미)'와 '기간 내 발생 횟수(totalCount, 1시간 버킷 누적 합산)' 둘로 명확히 분리. detect/timeline/summary 도메인의 'user' 식별자를 'node' 의미로 일관 통일(OR-1355 — 22커밋), manager-web의 '사용자/부서/PC명/도메인명' 4필드 표시를 source-of-truth 정리(OR-1294)까지.

**Playwright MCP + v3 비동기 I/O (OR-1361, OR-1422)**. Claude Code 등 AI 에이전트가 manager-web의 로그인·페이지 진입·할당 흐름을 자연어 한 번으로 검증하도록 `@playwright/mcp` 3채널(headed/headless/legacy-v105)을 등록. 고객사 구형 Chromium v105 호환성 회귀까지 같은 명령으로 검증. v3 파일 업로드/다운로드/삭제 경로의 sync `os.path` / `open` / `os.remove` 호출을 `aiofiles.os` 기반 비동기 syscall로 전환해 '소수 파일 경로의 sync syscall이 다수 Mongo/Redis 요청의 p99 latency까지 오염시키는 비대칭 구조'를 차단.

**WebSocket 핸들러 안전화 (OR-1432, OR-1433)**. `/v3/ws/node/stats` 토큰 누락/만료 시 `AttributeError: 'WebSocket' object has no attribute 'method'`로 ASGI 핸들러가 2차 폭발하던 버그. ## SungWoo824 — RBAC 권한 체계 + Baseline 일상 분류 서비스

5월 한 달 94커밋. 권한 체계와 baseline 신규 서비스 두 큰 라인을 동시에 굴렸다.

**관리자 계정 권한(RBAC) 시스템 — Store + Redis 캐시 + 위젯 ACL 동기화 (OR-1314)**. `/v3/acl` 통합 라우터(endpoint/widget/feature/role 14개 엔드포인트)와 Store 컬렉션 패턴(`EndpointAclStore` · `WidgetAclStore` · `RoleDefinitionStore` · `FeaturePermissionStore`) + Redis 캐시를 깐 위에서, 5월에는 `_check_endpoint_acl`에 `_OPEN_ACCESS_OPERATIONS` 바이패스 추가, 위젯 ACL R/W/D 동기화 버그 수정, role 삭제 시 `FeaturePermissionStore` 캐스케이드 정리, HTTPException → 커스텀 예외 전환, 로그인 응답 ACL 직렬화 camelCase 정합까지 안정화 마무리.

**LiveCMD Role별 권한 적용 — Agent ACL 확인 엔드포인트 + 403 상세 응답 (OR-1415)**. Agent용 LiveCMD 권한 확인 엔드포인트를 신설하고, `feature_permission` 빈 배열을 전체차단으로 정의 + 서버 시작 시 기본값 생성. 403 응답에 거부된 method/path/operation_id를 동봉해 매니저의 `PermissionErrorBoundary` 오버레이가 어떤 API가 막혔는지 정확히 표시. 기능 권한에서 batch_command(1:N) 항목 제거 + ACL 직렬화 camelCase 정합.

**새 엔드포인트 추가 알림 — is_reviewed 추적 + 토스트 + 미확인 배지 (OR-1359)**. 서버 배포 후 `acl_seed`가 새로 시드한 엔드포인트를 `is_reviewed=False`로 추적하고, admin 로그인 시 `useUnreviewedEndpointAlert` 훅이 sessionStorage 기반으로 세션당 1회 토스트 알림. '설정 > 권한 관리 > API 권한' 탭에 미확인 카운트 오렌지 배지, PUT 저장 시 `endpoint_update_case`가 자동으로 `is_reviewed=True` 설정, '모두 확인 완료' 일괄 처리 엔드포인트까지.

**설정 영역을 위젯 → 기본 내장으로 — ACL OR 합산 버그 근원 해소 (OR-1413)**. 설정(Setting)을 GridLayout의 위젯 배치에서 빼내고 `Content.tsx`가 setting 섹션을 직접 렌더링하도록 변경(`containType='global'`, 위젯 헤더바·드래그 핸들 없음). 위젯 패널 노출과 `widget.config.json`의 `requiredApis` 25개를 비워, 부모 setting 위젯과 자식 탭 위젯이 같은 API를 중복 보유해 `_sync_endpoint_acl`의 OR 합산으로 권한 박탈이 되돌려지던 버그를 근본 차단.

**매니저 권한 적용 오류 수정 — 위젯별 403 + 타입 안전성 + 보안 (OR-1402)**. `throwOnError`에서 `PermissionDeniedError`만 선택적으로 throw하도록 바꾸고, `PermissionErrorBoundary`를 `DynamicComponentLoader` 레벨(위젯 부모)로 옮겨 위젯 단위 오버레이로 캐치. `aclTypes.ts` 신설로 `useQuery` 5개에 제네릭 타입 적용, `roleId`의 `dangerouslySetInnerHTML` 제거(XSS 방지), `useUnreviewedEndpointAlert`에 `AbortController` 추가.

**Baseline 일상 분류 서비스 — Sprint 1+1.5: 5조건 룰 + 다도메인 + LightGBM-ready 스키마 (OR-1438)**. detect를 시작으로 system / process / nodeinfo까지 확장될 "늘 똑같이 반복되는 일상 노이즈" 자동 분류 독립 서비스. 1순위 요구사항을 "기존 데이터 / 서비스에 영향 0"으로 잡고 원본 컬렉션을 read-only(`find`만)로 두며 쓰기는 자체 컬렉션(`baseline_routine` · `baseline_overrides` · `baseline_runs`)에만 upsert, 도메인별 독립 systemd unit(`baseline-detect.service`, ...)로 한 도메인 장애가 다른 도메인·기존 서비스에 영향 없도록 격리.

'일상' 판정은 5조건이 모두 만족할 때만 routine. `cv_count < 0.15`(발생량이 일정) · `k ≥ 3`(서로 다른 3일 이상 발생) · `max_gap ≤ 14일`(2주 넘게 끊긴 적 없음) · `since ≤ 14일`(최근 2주 안에 발생 = 살아있음) · `rep ≥ 0.6`. 핵심 설계는 반복성 점수 `rep = max(reg, freq)` — 규칙성 `reg = 1 − min(gap_cv, 1)`과 빈도 `freq = min(1, 2 ÷ mean_gap)`를 max로 조화해서 "규칙적으로 반복"과 "자주 반복(들쭉날쭉해도)" 두 형태의 일상을 모두 포착. 다섯 조건 모두 사람이 검산 가능한 단순 통계라 운영자가 결정 근거를 그대로 본다.

운영 서버 detect_node 전체 이력(186,833행 · 206일)로 공식 도출 · 검증. 초기 시도였던 EWMA+CUSUM은 단순 cv 규칙 대비 가치가 입증되지 않아 폐기. 5월 마지막 주에는 운영 hang 수정(`fork → spawn` 전환), `baseline settings`를 `store.code=baseline` 패턴으로 통합, multiprocessing 벤치마크(`chunksize 50`이 기본, `cpu_count // 2` 워커가 14 코어 머신의 sweet spot)까지 Sprint 1.5 검증 마무리.
